省流一句话:
Agent Memory 继续照搬 RAG范式 的“相似度 top-k”会天然取回重复的证据链，会混淆模型或者找不到真正的答案。我们把记忆从传统“chunk 索引”解耦成“语义组件索引”，再配套一个自上而下的检索，让回答更准、token 更省。（详情请关注我们上一个帖子!）
	
现状: 为什么 Agent Memory 和传统 RAG 不一样？
不少Memory 框架仍是标准RAG：
chunks → 向量化 → top-k → 拼上下文 → LLM 回答
但 RAG 面向异质文档库，主要风险是“检索到不相关”；
而 Agent Memory 是一条连贯对话流，候选普遍都“相关”，问题变成：
近重复太多：top-k 很容易塌缩在同一团相似区（全是“差不多的证据”）
后处理容易误伤：pruning/压缩会剪断跨轮次证据链（指代/省略/时间依赖一断就错）
所以矛盾不是“找不到相关”，而是：取回太重复 + 剪枝容易删错前提。
我们怎么做（核心两件事）
(A) 改索引：chunk → 语义组件索引
把对话记忆拆成“语义组件”，再组织成更高层结构；构建时用一个 稀疏度–语义一致性 的目标来指导拆分/合并，让结构既不拥挤也不碎。
(B) 改检索：一次性 top-k → 自上而下逐级取证
先选一个精但全的高层骨架，再只在“能提升模型确定性”时扩展细节。
直观效果：少取重复、不剪碎链条。
在 LoCoMo / PerLTQA、以及多种开源/闭源 LLM 上，我们看到：回答更稳, token 更省。
关键不是“检索完再剪”，而是从源头换索引与检索范式。
