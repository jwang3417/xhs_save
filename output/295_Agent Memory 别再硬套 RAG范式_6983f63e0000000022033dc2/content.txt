Beyond RAG for Agent Memory: Retrieval by Decoupling and Aggregation
省流一句话：
Agent Memory 继续照搬 RAG范式 的“相似度 top-k”会天然取回重复的证据链，会混淆模型或者找不到真正的答案。我们把记忆从传统“chunk 索引”解耦成“语义组件索引”，再配套一个自上而下的检索，让回答更准、token 更省。
1）现状：很多 Memory Agent 框架越用越贵、还不一定更准？
不少 memory agent 仍是标准 RAG：chunks → 向量化 → 相似度 top-k → 拼上下文 → LLM 回答。
但“前提错位”：RAG 面向异质文档库，主要翻车是检索到不相关；而 Agent Memory 是有边界、连贯对话流，候选高度相关且近重复多。
→ 固定 top-k 易塌缩到同一密集相似区，取回“很相似”却未必“真正需要”（冗余）。
→ 冗余后常加 pruning/compression 等后处理，但对话证据有指代/省略/时间依赖，常是一条跨轮次证据链；剪枝易删前置条件，链断就答错。
所以矛盾不是“找不到相关”，而是：top-k 取回冗余；后处理又可能删错关键前提。
2）动机：问题不在“再找更像”，而在“别让检索被 raw span 相似度牵着走”。
换目标：span matching → 组件级证据选择（component-level），让冗余被结构性避免。
核心范式：Decoupling→Aggregation
→ 拆成语义组件（latent components）→ 组织成高层结构 → 用结构反向驱动检索：不再由 raw span top-k 排名决定。
直观：embedding 很像的两段，只要在不同组件里也不易一起取回；冗余不是靠后剪，而是结构层面规避。
结构要求：组件够区分、聚合够忠实。
3）我们怎么做
(A) 改索引：chunk → 语义组件索引（语义点+完整证据链）；构建时用“稀疏度–语义一致性”目标指导主题拆分/合并，让结构不拥挤也不碎。
(B) 改检索：一次性 top-k → 自上而下逐级取证；先选精但全的高层骨架，再仅在“能提升模型确定性”时扩展细节，少取重复、不剪碎链条。
4）结果&结论
LoCoMo/PerLTQA、跨开源/闭源 LLM 验证：整体回答更稳、token 更省。关键不是“检索完再剪”，而是从源头换索引与检索方式。
