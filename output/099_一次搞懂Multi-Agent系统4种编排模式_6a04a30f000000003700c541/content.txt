Multi-Agent系统主流4种编排模式：
	
1️⃣中心化编排者-工作者模式 (Orchestrator-Workers)
这是最经典、使用最广泛的树状/星状结构。一个核心的主管 Orchestrator负责全局控制，动态拆解任务并分发给多个员工Workers，最后将结果汇总返回。
工作流程： 主管接收输入 -> 拆解子任务 -> 派发给专用子 Agent -> 收集各子 Agent 结果 -> 统一聚合输出。
适用场景： 复杂的数据分析报告生成等需要中央审计和强控制的场景。
优缺点： 可控性与透明度极高，但主管 Agent 容易成为系统性能和 Token 消耗的瓶颈。
	
2️⃣路由/转交模式 (Routing / Hand-off)
一种基于状态机或分类器的拓扑结构。一个前端 Agent 充当路由分析用户的意图，然后将任务完全移交给最匹配的下游专业 Agent。
工作流程： 用户输入 -> 路由 Agent 分类判断 -> A -> A 直接与用户交互（或直接交班给 B ）。
适用场景： 企业多功能客服系统
优缺点： 极高的高并发能力和低延迟；但 Agent 之间如果边界模糊，容易出现循环路由（死锁）。
	
3️⃣管道/顺序模式 (Pipeline / Sequential)
将任务组织成一条线性流水线。上一个 Agent 的输出直接作为下一个 Agent 的输入，每个 Agent 只负责一个固定的特定阶段。
工作流程： 任务 -> Agent A-> Agent B-> Agent C-> 最终输出。
典型场景： 自动化代码生成流水线。优缺点： 逻辑完全确定，结果高度可预测；缺点是缺乏弹性，若前序步骤出错，错误会链式放大。
	
4️⃣群聊/网络协作模式 (Group Chat / Swarm / Mesh)
一种去中心化的编排模式。所有 Agent 加入一个公共的“通信总线”或对话池中，根据当前对话状态自主竞争、发言或辩论，以协同得出最佳结论。
工作流程： 任务发布到公共群聊 -> Agent 们自由讨论/互相质询 -> 达成共识后输出。
典型场景： 多专家联合会诊
优缺点： 创新性和涌现性最强；但极难控制，容易产生无限循环对话，导致 Token 消耗爆炸。
