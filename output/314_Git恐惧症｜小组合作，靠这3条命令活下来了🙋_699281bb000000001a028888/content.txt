Git恐惧症｜小组合作，靠这3条命令活下来了🙋

CS留学生们，你是不是也这样：
第一次用Git小组合作，听到最多的话就是“代码被我冲没了”、“冲突不会解”、“不敢pull”。
作为一个被Git折磨过一整个学期的过来人，我想说：你不是一个人。
	
其实Git没那么可怕，你只需要先搞懂一件事：“我在哪”和“怎么去”。
	
🗺️ 一张图看懂Git的“四层空间”
	
工作区：你正在写代码的地方（VS Code里看到的）
暂存区：打包站，用add把文件放进去
本地仓库：你电脑里的时光机，每次commit就是一次存档
远程仓库：云端的代码库，所有人最终都往这儿推
	
流转公式： 工作区 → add → 暂存区 → commit → 本地仓库 → push → 远程仓库
（配图：四层空间流转图）
	
✅命令一：别再直接pull了，学会“先看后买”
新手最爱干的事：
git pull，然后代码没了、冲突炸了、人傻了。
	
👉真相：pull = fetch + merge，拆开用才是高手
👉正确姿势：见图
	
遇到冲突怎么办？
首先识别冲突长什么样：见图
👉处理原则：留下对的，删掉标记（<<<<<<<、=======、>>>>>>>），然后：
git add .
git commit
	
✅命令二：后悔药——reset和revert怎么选？
	
👉场景A：刚commit完就后悔（还没push）
用reset，三个模式区别很大：见图
	
👉场景B：已经push到远程了
千万别用reset，队友会恨你😂
用revert——创建一个“反向提交”来抵消错误：见图
👉为什么安全：不改历史，队友拉代码时能看到清晰的撤销记录。
	
✅命令三：stash——写到一半被叫去修bug的救命神器
👉场景：代码写一半（不能commit），组长说有个紧急bug要修。
👉操作：见图
如果取出来时有冲突：手动解决，然后git add .就行。
	
给新手的最后建议：
✅ 小提交：别攒一周代码才commit。完成一个小功能就commit一次。这样翻车也只会丢一小段。
	
✅ 勤拉取：每天开工前git fetch一下，看看队友进度。越早解决冲突，痛苦越小。
	
✅ 认真写备注：别只写“update”。写清楚改了啥，这是团队协作的第一步。
